## 🎯 1. الملخص التنفيذي

منصة SchoolOS في مدرسة الشحانية تمتلك **بنية تقنية سليمة ومتكاملة** تتضمن هيكلاً هرمياً واضحاً للخطة التشغيلية (مجال → هدف → مؤشر → إجراء → دليل) ونظام لجان مدروساً. غير أن التحليل المعمق للبيانات والكود يكشف عن **فجوة تنفيذية حرجة** وإشكاليات بنيوية في منظومة اللجان تستوجب التدخل الفوري.

### النقاط الجوهرية

| المؤشر | القيمة | التقييم |
|--------|--------|---------|
| إجمالي الإجراءات | 2,896 | — |
| الإجراءات المكتملة | 317 | 🔴 11% فقط |
| الإجراءات قيد التنفيذ | 2,579 | 🔴 89% متعثرة |
| عدد المجالات | 7 | ✅ |
| عدد الأهداف | 33 | ✅ |
| المنفذون المسجلون | 32 | ✅ |
| أعضاء لجنة المراجعة | 10 | ✅ |
| اللجنة التنفيذية — واجهة مستخدم | لا يوجد | 🔴 حرج |

> **التحذير الأهم:** مع اقتراب نهاية العام الدراسي (مايو 2026)، فإن بقاء 89% من الإجراءات دون إنجاز يمثل خطراً مباشراً على التقييم المؤسسي للمدرسة.

## 🔍 4. تحليل السواط SWOT

### نقاط القوة (Strengths) ✅

| # | نقطة القوة | التفاصيل |
|---|------------|---------|
| 1 | **هيكل هرمي سليم** | مجال → هدف → مؤشر → إجراء → دليل — تصميم أكاديمي متكامل |
| 2 | **ربط المنفذين** | نموذج `ExecutorMapping` يتيح ربط المسمى الوظيفي بمستخدم حقيقي |
| 3 | **لجنتان تنظيميتان** | لجنة تنفيذية ولجنة مراجعة ذاتية مخصصتان |
| 4 | **تغطية شاملة** | 7 مجالات تغطي جميع جوانب العملية المدرسية |
| 5 | **نظام الأدلة** | `ProcedureEvidence` مع رفع ملفات مرتبطة بكل إجراء |
| 6 | **33 هدفاً موزعاً** | ترقيم منظم وتوزيع واضح على المجالات |
| 7 | **دعم متعدد السنوات** | حقل `academic_year` يتيح أرشفة الخطط السابقة |
| 8 | **تسجيل تلقائي للتواريخ** | `created_at` و `updated_at` على كل إجراء |
| 9 | **فلترة متقدمة** | إمكانية تصفية الإجراءات بالحالة والمنفذ والمجال |
| 10 | **حل N+1** | استخدام `Prefetch` في `domain_detail` view لتحسين الأداء |

### نقاط الضعف (Weaknesses) 🔴

| # | نقطة الضعف | الخطورة |
|---|------------|---------|
| 1 | **89% من الإجراءات غير مكتملة** — 317 فقط من 2,896 | 🔴 حرجة |
| 2 | **اللجنتان غير مترابطتين في الكود** — نموذجان منفصلان | 🔴 حرجة |
| 3 | **ربط executor يدوي** — مصدر تناقض دائم بين النص والـ FK | 🔴 حرجة |

| 5 | **صلاحيات اللجنة التنفيذية على مستوى اللجنة لا الفرد** | 🟠 عالية |
| 6 | **لا توجد views/URLs للجنة التنفيذية** — نموذج بلا واجهة | 🟠 عالية |
| 7 | **تكرار في executor_norm** — اختلافات إملائية تلوّث البيانات | 🟠 عالية |
| 8 | **صف مفقود في quality_structure.csv** — عضو بلا مسمى وظيفي | 🟡 متوسطة |
| 9 | **date_range نصي** — لا يمكن فرض مواعيد آلياً | 🟡 متوسطة |
| 10 | **لا يوجد approval workflow** — الإجراءات تُكتمل دون مراجعة | 🟡 متوسطة |

### 6.1 المقارنة البنيوية

#### نموذج `OperationalPlanExecutorCommittee` — لجنة المنفذين

```python
class OperationalPlanExecutorCommittee(models.Model):
    school        = FK → School           # ✅ صحيح
    academic_year = CharField(9)          # ✅ صحيح
    name          = CharField(100)        # ✅ صحيح
    members       = M2M → CustomUser      # 🔴 بلا أدوار للأعضاء
    can_execute_plan  = BooleanField      # 🔴 على اللجنة كلها لا الفرد
    can_review_plan   = BooleanField      # 🔴 نفس المشكلة
    can_report_progress = BooleanField    # 🔴 نفس المشكلة
    
    # unique_together = (school, academic_year)
    # 🔴 لجنة واحدة فقط لكل مدرسة/سنة — مقيّد جداً
    
    # Views: ❌ لا يوجد
    # URLs:  ❌ لا يوجد
    # Template: ❌ لا يوجد
```

#### نموذج `QualityCommitteeMember` — لجنة المراجعة الذاتية

class QualityCommitteeMember(models.Model):

    job_title      = CharField(100)       # ✅ مسمى وظيفي واضح
    responsibility = CharField choices   # ✅ رئيس/نائب/مقرر/عضو
    domain         = FK → Domain         # 🟡 خطر عند حذف المجال
    academic_year  = CharField(9)        # ✅ صحيح
    is_active      = BooleanField        # ✅ إمكانية التعطيل
    
    # committee_type: ❌ غير موجود — لا يمكن التمييز عن اللجنة الأخرى
    
    # Views: ✅ موجودة
    # URLs:  ✅ موجودة
    # Template: ✅ موجود

## ⚠️ 7. المشكلات الحرجة المكتشفة

### المشكلة #1 — انفصام معماري بين النموذجين 🔴 حرجة

**الوصف:**
`OperationalPlanExecutorCommittee` و `QualityCommitteeMember` نموذجان مستقلان تماماً في قاعدة البيانات. لا يوجد FK أو M2M يربطهما. عضو لجنة المراجعة الذاتية لا يمكن ربطه بلجنة المنفذين برمجياً.

**الدليل التقني:**
النموذج الأول وُضع في **بداية ملف `models.py`** قبل الـ docstring وقبل تعريف الفئات الرئيسية — مما يدل أنه أضيف لاحقاً كـ *patch* وليس تصميماً أصلياً.

**الأثر:** استحالة بناء تقرير موحّد يجمع أداء المنفذين مع تقييم لجنة المراجعة.

# management command: normalize_executors
class Command(BaseCommand):
    help = 'Normalize executor_norm values to fix spelling variations'
    
    NORMALIZATION_MAP = {
        'منسق التربيه الاسلاميه':   'منسق التربية الإسلامية',
        'منسق اللغه العربيه':        'منسق اللغة العربية',
        'منسق اللغه الانجليزيه':     'منسق اللغة الإنجليزية',
        'منسق الفنون البصريه':       'منسق الفنون البصرية',
        'منسق التربيه البدنيه':      'منسق التربية البدنية',
        'منسق الدراسات الاجتماعيه': 'منسق الدراسات الاجتماعية',
        'النائب الاكاديمي':          'النائب الأكاديمي',
    }
    
    def handle(self, *args, **options):
        for old, new in self.NORMALIZATION_MAP.items():
            count = OperationalProcedure.objects.filter(
                executor_norm=old
            ).update(executor_norm=new)
            if count:
                self.stdout.write(f'✅ {old} → {new}: {count} إجراء')
        
        # تحديث ExecutorMapping أيضاً
        for old, new in self.NORMALIZATION_MAP.items():
            ExecutorMapping.objects.filter(
                executor_norm=old
            ).update(executor_norm=new)
```

def executor_committee_detail(request):

    year   = request.GET.get("year", "2025-2026")

    committee_members = QualityCommitteeMember.objects.filter(
        school=school,
        academic_year=year,

        is_active=True,
    ).select_related("user", "domain")
    
    # إجراءات كل عضو
    member_stats = []
    for member in committee_members:
        if member.user:
            procs = OperationalProcedure.objects.filter(
                school=school,
                executor_user=member.user,
                academic_year=year,
            )
            member_stats.append({
                "member":    member,
                "total":     procs.count(),
                "completed": procs.filter(status="Completed").count(),
                "pending":   procs.filter(status="In Progress").count(),
            })
    
    return render(request, "quality/executor_committee.html", {
        "committee_members": committee_members,
        "member_stats":      member_stats,
        "year":              year,
    })

from celery import shared_task
from datetime import date, timedelta

@shared_task
def notify_upcoming_deadlines():
    """يُرسل تنبيهاً للمنفذ قبل 7 أيام من الموعد النهائي"""
    target_date = date.today() + timedelta(days=7)
    overdue = OperationalProcedure.objects.filter(
        deadline=target_date,
        status="In Progress",
        executor_user__isnull=False,
    ).select_related("executor_user", "indicator__target__domain")
    
    for proc in overdue:
        send_mail(
            subject=f"تذكير: إجراء [{proc.number}] موعده بعد 7 أيام",
            message=f"الإجراء: {proc.text}\nالمجال: {proc.indicator.target.domain.name}",
            recipient_list=[proc.executor_user.email],
        )

class CommitteeManager(models.Manager):
    def by_committee_type(self, school, year, committee_type):
        return self.filter(
            school=school,
            academic_year=year,
            committee_type=committee_type,
            is_active=True,
        )
    
    def search(self, school, year, name=None, role=None, domain=None):
        qs = self.filter(school=school, academic_year=year)
        if name:
            qs = qs.filter(
                Q(user__full_name__icontains=name) |
                Q(job_title__icontains=name)
            )
        if role:
            qs = qs.filter(responsibility=role)
        if domain:
            qs = qs.filter(domain=domain)
        return qs
```
